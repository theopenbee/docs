title: 数字员工
description: 通过对话创建和管理 AI 数字员工

数字员工是代你执行任务的 AI 代理。每个数字员工拥有自己的名称、描述、记忆和工作目录。你可以通过 Web 控制台或直接在 IM 平台中与 Bee（协调器）对话来管理数字员工。

## 通过对话管理数字员工

你可以通过自然对话来创建、更新和删除数字员工：

### 创建数字员工

> "创建一个叫小明的数字员工，帮我做代码审查"

> "我需要一个写文档的数字员工，叫文档助手"

Bee 会创建数字员工并向你确认。你可以在消息中指定名称、描述和初始记忆。

<BeeNote>
  描述字段是路由的核心 — Bee 全靠它来决定把任务交给谁。写得越具体，路由越准确，少踩「分配错了」的坑 🐝
</BeeNote>

### 更新数字员工

> "把小明的描述改为专注于 Go 代码审查"

> "修改文档助手的记忆，让它始终用中文写作"

### 删除数字员工

> "删除文档助手"

> "把小明这个数字员工删掉，把工作目录也清理了"

### 查看数字员工列表

> "列出所有数字员工"

> "我有哪些数字员工？"

### 查看数字员工状态

> "小明现在什么状态？"

> "查看系统状态"

## 数字员工属性

每个数字员工拥有以下属性：

| 属性 | 描述 |
|------|------|
| **名称** | 数字员工的显示名称，用于在对话中指名分配任务 |
| **描述** | 数字员工的职责 — Bee 根据描述进行任务路由 |
| **记忆** | 影响数字员工行为的持久化上下文 |
| **工作目录** | 数字员工操作的文件系统路径。默认为 `~/.openbee/worker/{id}`，也可以指定自定义绝对路径。 |

## Bee 如何将任务路由到数字员工

当你发送消息时，Bee 会按以下优先级决定由哪个数字员工处理：

1. **明确指定** — 如果你提到了某个数字员工的名字，任务直接分配给该数字员工。例如："小明，帮我审查这个 PR"
2. **对话延续** — 后续消息会分配给上一个任务的同一个数字员工。例如："再简洁一些"
3. **描述匹配** — Bee 会将你的请求与数字员工描述进行语义匹配
4. **兜底处理** — 如果没有匹配的数字员工，Bee 会建议你创建一个

## 最佳实践

### 编写清晰的描述

描述是任务路由最重要的字段。Bee 用它来将你的请求匹配到合适的数字员工。

**好的描述** — 具体、聚焦技能：
- "负责 Go 后端开发、代码审查和 API 设计"
- "编写和维护中英文项目文档"
- "监控系统健康状况并生成每日状态报告"

**差的描述** — 模糊、无法有效路由：
- "数字员工1"
- "做事情"
- "通用助手"

当多个数字员工的描述存在重叠时，Bee 会询问你选择。保持描述各不相同，这样路由就能自动完成。

### 用记忆塑造行为

记忆会写入数字员工的系统提示词，在所有任务中持久生效。用它来定义数字员工的个性、专业知识和约束。

**领域专业知识：**
> "你是一个经验丰富的 Python 开发者。始终遵循 PEP 8 规范，优先使用类型注解。"

**沟通风格：**
> "用简短的要点回复，避免长篇大论。"

**约束与安全：**
> "在没有和用户确认之前，绝不进行破坏性的 API 变更。"

**上下文与受众：**
> "你在帮助一个非技术用户，用简单易懂的方式解释一切。"

你可以随时通过对话更新记忆：

> "更新小明的记忆，加上：提交代码前必须运行测试"

<BeeNote>
  记忆会直接写进数字员工的系统提示词，每次任务都生效。如果给同一个数字员工设了矛盾的指令（比如「简短回复」和「详细解释每一步」），它会乱的 — 定期检查一下记忆内容是个好习惯 🐝
</BeeNote>

### 为项目设置自定义工作目录

默认情况下，每个数字员工会获得一个独立目录 `~/.openbee/worker/{id}`。如果数字员工需要操作特定项目，将工作目录设置为你的项目路径：

> "创建一个叫 DevBot 的数字员工，工作目录设为 /Users/me/my-project"

这样数字员工就可以直接读取、编辑和管理你项目中的文件。工作目录必须是绝对路径。

### 使用简短易记的名称

给数字员工起简短好记的名字，方便你快速分配任务：

> "小明，修复登录 bug"

> "文档助手，更新 README"

当你在消息中提到数字员工名字时，Bee 会直接路由到该数字员工，不需要任何猜测。

### 创建专职数字员工

与其创建一个通用数字员工，不如为不同领域创建专职数字员工。这样可以提升：

- **路由准确性** — 具体的描述能更好地匹配请求
- **回复质量** — 聚焦的记忆能产生更相关的输出
- **任务隔离** — 每个数字员工维护自己的上下文和工作状态

**示例团队配置：**
- **开发小哥** — "后端开发、代码审查、API 服务器 bug 修复"
- **文档助手** — "技术文档、README 更新、API 参考编写"
- **运维管家** — "系统监控、部署检查、基础设施管理"

## CLAUDE.md 自定义

每个数字员工的工作目录中都有一个 `CLAUDE.md` 文件，用于定义数字员工的系统提示词。该文件由 OpenBee 根据数字员工的记忆字段自动管理，你也可以直接编辑它以进行高级自定义。

## 数字员工状态

| 状态 | 含义 |
|------|------|
| `idle` | 数字员工空闲，可以接受任务 |
| `working` | 数字员工正在执行任务。新任务会排队等待。 |
| `error` | 数字员工在执行过程中遇到错误 |
